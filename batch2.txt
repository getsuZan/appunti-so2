Programmazione di sistema
• Il Kernel
• componente del sistema operativo che gestisce le risorse disponibili
• offre l'accesso e l'utilizzo delle risorse da parte dei processi

• CPU Central Processing Unit
• RAM Random Access Memory
• I/O I dispositivi di Input e Output

• Il numero varia con la versione del sistema: UnixV7 50, 4.4BSD 110, Linux 380, …

• Permettono ad un processo di accedere ai servizi offerti dal kernel
• quindi … permettono al programmatore di creare programmi che interagiscono
direttamente con il kernel

Gestione errori system call
• L'esecuzione di una system call può interrompersi e non andare a
buon fine per diversi motivi, tra cui principalmente:
• il processo che la invoca non ha sufficienti privilegi per l'esecuzione
• non ci sono sufficienti risorse per l'esecuzione
• gli argomenti in ingresso alla system call non sono validi

• E’ fondamentale controllare i valori di ritorno per rilevare e segnalare
all’utente il verificarsi di errori.
• Per un corretto funzionamento del programma è FONDAMENTALE
GESTIRE IN MANIERA OPPORTUNA l’eventuale errore verificatosi
zuliani@di.uniroma1.it

errno
• Le system call che terminano con un errore
• tipicamente ritornano il valore -1 (controllare caso per caso!!)
• impostano la variabile globale errno con il codice specifico dell'errore che si
è generato durante l'esecuzione

• Le system call che terminano con successo lasciano invariato errno
• La variabile errno rappresenta il codice di errore dell’ultima system
call invocata che ha generato un errore
• Ha senso utilizzare tale variabile solo dopo essersi accertati che la system call
ha ritornato un valore di errore (-1). Altrimenti si rischia di considerare il
valore di errno inconsistente.
zuliani@di.uniroma1.it

perror()
• funzione della libreria standard
#include <stdio.h>
void perror(const char *prefix);

• stampa su stderr il messaggio di errore convertendo il codice di
errore errno in una stringa formata da
<prefix>:<errno_string>
• <errno_string> rappresenta il messaggio di errore in formato di stringa (e
quindi mnemonico) associato al valore di errno.
• ad es. perror(“main”); invia su stderr la stringa
“main:messaggio_errore_mnemonico_=_errno”

void *realloc(void *ptr, size_t size)
• modifica la dimensione dell’area di memoria precedentemente allocata con
m/calloc e puntata da ptr nella dimensione specificata dal valore di
size
• Ritorna NULL in caso di errore
• ad es. (da es. precedente)
strptr1=(char *) realloc(strptr, 10*SIZE_OF_ARRAY);

• strptr1 potrebbe essere diverso da strptr infatti, nel caso di
aumento della dimensione!
• Se realloc non riesce ad allargare l’area correntemente allocata e
puntata da ptr, allora quest’ultima viene liberata e una nuova area
allocata e ritornata al chiamante.

realloc (cont.)

• La nuova area di memoria allocata NON viene inizializzata (se ad
esempio quella originale lo era)
• In caso di errore (ad esempio non c'è memoria a sufficienza) l’area di
memoria originale rimane intatta.

Memory leakage («perdita» di memoria)
• L'esecuzione di un programma che non gestisce correttamente la
liberazione della memoria non più utilizzata, può causare un aumento
del consumo della memoria del sistema. Questo può portare al
fallimento del programma stesso, non riuscendo più ad allocare altra
memoria da utilizzare, ed in generale può portare al deterioramento
delle prestazioni e del funzionamento del sistema.
• Usare sempre free() dopo che avete terminato l’uso della
memoria allocata dinamicamente

Sync e de-mapping
#include <sys/mman.h>
int msync(void *addr, size_t len, int flags);
• Le modifiche ad una regione mappata (con MAP_SHARED, ovviamente) non vengono
scritte immediatamente nel file – il kernel decide quando farlo
• Con msync comunicate al kernel che volete scrivere sul file le modifiche in memoria

#include <sys/mman.h>
int munmap(void *addr, size_t len);
• Quando il processo termina la memoria viene automaticamente de-mappata, ma ciò
non significa che il contenuto è scritto sul file (dipende dalla politica di gestione della
memoria virtuale)
• Ciò non avviene se chiudo il file, quindi sempre buona regola de-mappare la
memoria con munmap

Operare sui file
• File, astrazione che descrive ogni risorsa, ad es. file su disco,
dispositivi, socket, fifo
• Sequenza tipica di azioni
1. apertura file mediante open; viene creato un file descriptor
2. operazione su file mediante syscall; e.g. write, read
3. chiusura del file mediante close

• Quando un processo termina, chiude automaticamente tutti i file
correntemente aperti, tuttavia, è sempre consigliabile e buona regola
di programmazione chiudere esplicitamente un file invocando la
close
zuliani@di.uniroma1.it

File descriptor
• E’ il riferimento ad un file aperto: è un intero
piccolo, generato sequenzialmente a partire
dal valore 0. Per default, ogni processo, ha
associato i seguenti file descriptors:
• 0; stdin
• 1; stdout
• 2; stderr

• Quando un file viene chiuso, il suo file
descriptor viene liberato e può essere
riutilizzato. È possibile aprire uno stesso file
ed ottenere file descriptor diversi che
puntano allo stesso file (e.g., due utenti
aprono lo stesso file in lettura).
zuliani@di.uniroma1.it

5

File flags associati ad un File descriptor
• File status flags:
• Associati allo stato di un file
• Sono condivisi tra tutti i file descriptor ottenuti per duplicazione da un unico
file descriptor.

• File descriptor flags:
• Associati al file descriptor
• Indipendenti dal contenuto o status del file
• Descrivono proprietà e comportamento delle operazioni effettuate sul file
• File descriptor diversi che fanno riferimento ad uno stesso file hanno ognuno
il proprio insieme di descriptor flags
• Alcuni definiti solo per alcuni tipi di file speciali (es: fifo socket)
zuliani@di.uniroma1.it

Rappresentazione dei Flag
• Mediante maschere di bit
• se il bit è a 1 (settato) il flag è settato

• Per ogni flag è stata definita una maschera di bit o MACRO
• ad es. O_RDONLY

• I flag possono essere combinati mettendo in OR bit a bit (bitwise) le
maschere
• ad es.
• MACRO1 = 01000000;
• MACRO2 = 00010000
• MACRO1 | MACRO2 = 01010000
• dopo l’OR entrambi i flag sono settati

File status flags: 3 categorie
• Modalità di accesso
• read, write, read+write.
• specificati nella open
• non possono essere modificati una volta aperto il file

• Di apertura

• Modalità operative
• definiscono il comportamento delle operazioni read e write.

• the size of the buffer,
• a count of the number of characters currently in the buffer,

• and the like.

Pipe e FIFO
• Unix ha due tipi di IPC: FIFO (o named pipe) e pipe (o unnamed pipe)
• La scrittura dei dati su una pipe/fifo e quindi anche la lettura avviene in
maniera sequenziale (First In First Out)
• La FIFO è uno speciale tipo di file che può essere creato dalle system call
mkfifo o mknod. Può essere utilizzata da più processi, full-duplex
• La pipe è una struttura dati in memoria, è half-duplex. La creazione della
pipe può essere effettuata con la system call pipe che crea due file
descriptor (uno in lettura ed uno in scrittura)
• Esempio: un processo crea una pipe, poi crea un figlio e usa la pipe per
comunicare con il figlio che eredita i descrittori dei file
• La pipe può anche essere usata tra 2 processi che hanno un antenato
comune
zuliani@di.uniroma1.it

• Se tutto OK, pipe crea due file descriptors e ritorna 0
pipefd[0]: file descriptor di lettura
pipefd[1]: file descriptor di scrittura

• In caso di problemi, pipe ritorna -1 e imposta errno (pipefd non
viene modificato)

int pipe(int pipefd[2])
I dati scritti sulla pipe sono «bufferizzati» dal kernel finché non sono
letti. Le pipe hanno una dimensione massima:
• Se un processo legge da una pipe vuota allora rimane bloccato
• Se un processo scrive su una pipe piena allora rimane bloccato
• Una pipe viene chiusa quando tutti e due i processi hanno invocato la close
• Operazioni di lettura (read) su una pipe il cui fd di scrittura è stato chiuso
con close ritorna 0
• Operazioni di scrittura (write) su una pipe il cui fd di lettura è stato chiuso
con close ritornano -1 e ricevono il segnale SIGPIPE

• vedi: pipe.c, pipequater.c e pipe_quintes.c

int mkfifo(const char *pathname,
mode_t mode)
• Crea una FIFO con nome pathname e modalità di accesso mode.
• Una volta creata una FIFO, può essere acceduta da qualsiasi processo
(che abbia i diritti per l’accesso al file) in lettura/scrittura, tuttavia, le
operazioni devono essere simultanee. Un processo che apre la FIFO in
lettura rimane bloccato finché non c’è nessun processo che la apre in
scrittura (e viceversa).
• mkfifo ritorna 0 in caso di successo e -1 in caso di errore
impostando errno di conseguenza.
• Una volta creata, la FIFO può essere gestita (open, read write e close) come
un qualsiasi altro file
• È possibile aprire una FIFO in maniera non bloccante passando l’attributo
O_NONBLOCK alla system call open

Race condition
• Se due o più flussi di esecuzione hanno una struttura dati condivisa, la
concorrenza dei flussi può determinare uno stato della struttura non
coerente con la logica di ciascuno dei flussi
• Race condition
• Lo stato della memoria condivisa tra due o più flussi di esecuzione concorrenti
dipende dall’ordine esatto (temporizzazione) degli accessi alla memoria stessa

• Le race conditions sono tra gli errori del software più insidiosi e difficili da
determinare:
• non sono per loro natura deterministici
• possono non apparire mai sul sistema di sviluppo
• possono non essere facilmente replicabili
• sono di difficile comprensione
zuliani@di.uniroma1.it

Esempio race condition
• Flusso #1 e Flusso #2 condividono una
variabile counter
• L’incremento e decremento di counter
non sono operazioni atomiche (indivisibili)!
• Ciascuna consiste nel:
• Trasferimento del valore di counter in un
registro (load)
• Operazione di incremento o decremento sul
registro
• Trasferimento del valore del registro in
counter (store)

Regione critica
• Una sequenza di istruzioni di un flusso di esecuzione che accede ad

Semaforo
• Inventato da E. W. Dijkstra negli anni ‘60
• Basato su una variabile intera contatore
• inizializzata con valore n > 0
• Accessibile tramite due primitive atomiche:
• Wait: attende che il contatore sia positivo, poi lo decrementa
• Signal: incrementa il contatore

• Semaforo contatore
• n > 1 può essere acquisito da n flussi di esecuzione in modo concorrente

• Semaforo binario (Mutex - Mutual exclusion)
• n = 1 può essere acquisito da un solo flusso alla volta

int pthread_mutex_init(pthread_mutex_t *mutex,
const pthread_mutexattr_t *mutexattr);
• Inizializza un mutex e imposta gli attributi a mutexattr
• Gli attributi determinano il comportamento del semaforo quando il
thread invoca un lock/unlock più volte consecutivamente
• Se mutexattr=NULL viene impostato il valore di default (fast)
altrimenti può essere specificato recursive o error checking
• fast

• un lock blocca il thread finché il lock precedente non è rilasciato (può creare stallo);
• unlock rilascia il semaforo e ritorna subito

• recursive

• permette allo stesso thread di mettere più lock (un contatore tiene conto del numero di
lock messi);
• unlock decrementa il contatore; rilascia il semaforo quando il contatore è 0
• error checking
• genera un errore in caso il thread cerchi di mettere un lock su di un mutex sul quale già
detiene un lock;
• unlock ritorna errore se il thread non aveva fatto una lock precedentemente

Applicazioni multithread
• In una applicazione tradizionale, il programmatore definisce un unico
flusso di esecuzione delle istruzioni
• La CPU esegue istruzioni macchina in sequenza il flusso di esecuzione “segue” la
logica del programma (cicli, funzioni, chiamate di sistema, gestori di segnali. . . )
• Quando il flusso di esecuzione arriva ad eseguire una exit() l’applicazione termina

• Le applicazioni multithread consentono al programmatore di definire
diversi flussi (o «fili», «trame» (come in un tessuto)) di esecuzione
• Ciascun flusso di esecuzione condivide le strutture di dati principali dell’applicazione
• Ciascun flusso di esecuzione procede in modo concorrente ed indipendente dagli
altri flussi
• L’applicazione finisce solo quando tutti i flussi di esecuzione vengono terminati
zuliani@di.uniroma1.it

Modello “da molti a 1”
• Un’applicazione multithread è costituita da un singolo processo
• A diversi thread utente corrisponde un singolo thread kernel
• Un singolo processo tradizionale

• Il kernel non è coinvolto nella gestione dei flussi dell’applicazione
• L’applicazione (eventualmente usando una libreria di sistema) gestisce
autonomamente i thread utente:
• Organizza i vari flussi di esecuzione
• Salva e ripristina gli stack UM e i contesti

• Se un thread invoca una chiamata di sistema bloccante, il processo (e
quindi tutti i thread utente) vengono bloccati
• Impossibile sfruttare in modo implicito il parallelismo interno del
calcolatore
• Implementazione detta “a livello utente” o green threads

• Gestisce comunicazione e sincronizzazione

• Ciascun thread utente può invocare chiamate di sistema bloccanti

• L’applicazione sfrutta in modo implicito il parallelismo interno del

• Implementazione detta “a livello kernel” o native threads

Libreria Pthreads
• La libreria pthreads è definita dallo standard POSIX (IEEE 1003.1c)

pthread_attr_t
• scope PTHREAD_SCOPE_PROCESS
• New thread is unbound; it competes for resources with all the other threads in the process and in the system.

• detachstate PTHREAD_CREATE_JOINABLE
• Exit status and thread are preserved after the thread terminates.

• stackaddr NULL
• New thread has system-allocated stack address.

• New thread has system-defined stack size.

• New thread inherits parent thread priority.
